# 서브에이전트 실습 가이드: C 코드 병렬 리뷰

## 개요

3개의 서브에이전트가 **동일한 C 코드를 동시에 독립적으로** 분석한다.
서브에이전트의 핵심 장점인 **병렬 실행**을 직접 체험한다.

```
                   ┌─ [Sub A: 구조 분석]   ──┐
입력 C 코드 ──────▶ ├─ [Sub B: 버그 탐지]   ──┼──▶ 오케스트레이터 종합
(동시 전달)        └─ [Sub C: 스타일 검사] ──┘
```

### 순차 방식 vs 병렬 방식 비교

```
순차 (이전 가이드):           병렬 (이번 가이드):
A 완료 → B 시작 → C 시작      A, B, C 동시 시작
총 시간 = A + B + C           총 시간 = max(A, B, C)
각 에이전트가 이전 결과 의존   각 에이전트가 원본 코드만 의존
```

---

## 실습에 사용할 C 코드

파일: [`sample.c`](sample.c)

각 프롬프트의 `[sample.c 내용 붙여넣기]` 자리에 `sample.c` 내용을 복사해서 사용한다.

---

## 실습 구조

| 에이전트 | 담당 | 입력 | 다른 에이전트 의존 여부 |
|---------|------|------|----------------------|
| Sub A | 코드 구조 분석 | 원본 코드 | 없음 |
| Sub B | 버그/취약점 탐지 | 원본 코드 | 없음 |
| Sub C | 코딩 스타일 검사 | 원본 코드 | 없음 |
| 오케스트레이터 | 결과 종합 | A + B + C 결과 | A, B, C 모두 완료 후 |

> 핵심: Sub A, B, C는 서로의 결과를 기다리지 않는다.

## 단계 1: 오케스트레이터 프롬프트 (병렬 실행 지시)

아래 프롬프트를 Claude에게 입력한다.
이 프롬프트 하나로 3개의 서브에이전트가 동시에 실행된다.

```
당신은 C 코드 리뷰 오케스트레이터입니다.
아래 C 코드를 3명의 전문가에게 동시에 분석 요청하세요.

각 전문가는 독립적으로, 서로의 결과를 기다리지 않고 병렬로 분석합니다.

---
전문가 A: 코드 구조 분석가
- 함수 목록 (이름, 매개변수, 반환값)
- 각 함수의 역할 한 줄 요약
- 사용된 변수 및 자료구조
- 제어 흐름 (루프, 조건문)

전문가 B: 보안 취약점 분석가
- 발견된 취약점 목록 (심각도: HIGH / MEDIUM / LOW)
- 각 취약점의 위치 (함수명)
- 취약점 유형 (Buffer Overflow, Memory Leak 등)
- 왜 위험한지 한 줄 설명

전문가 C: 코딩 스타일 검사가
- Linux 커널 코딩 스타일 기준으로 검사
- 위반 항목 목록
- 함수명/변수명 네이밍 평가
- 가독성 개선 포인트
---

분석할 코드:
[sample.c 내용 붙여넣기]

3명의 분석이 모두 완료되면 결과를 종합해서 최종 리포트를 작성하세요.
```

## 단계 2: 각 서브에이전트 개별 실행 (비교 체험)

오케스트레이터 없이 각 에이전트를 **별도 대화창에서 동시에** 실행해본다.
실제 병렬성을 직접 체감할 수 있다.

```
출력:
1. 함수 목록 (이름, 매개변수, 반환값)
2. 각 함수의 역할 한 줄 요약
3. 사용된 변수 및 자료구조
4. 제어 흐름 (루프, 조건문)
5. 외부 의존성 (include, 라이브러리 함수)

```

```

출력:
1. 취약점 목록 (심각도: HIGH / MEDIUM / LOW)
2. 위치 (함수명)
3. 유형 (Buffer Overflow, Memory Leak, Off-by-one 등)

```

```
3. 들여쓰기 및 괄호 스타일
4. 주석 스타일
5. 가독성 개선 포인트

[sample.c 내용 붙여넣기]
```

## 단계 3: 오케스트레이터 종합 프롬프트

Sub A, B, C가 모두 완료된 후 아래 프롬프트로 종합한다.

```
당신은 코드 리뷰 오케스트레이터입니다.
3개의 전문 에이전트 결과를 통합해서 최종 리포트를 작성하세요.
새로운 분석은 추가하지 말고, 통합과 우선순위 정리만 수행하세요.

=== Sub A 결과 (구조 분석) ===
[Sub A 결과 붙여넣기]

=== Sub B 결과 (취약점 분석) ===
[Sub B 결과 붙여넣기]

=== Sub C 결과 (스타일 검사) ===
[Sub C 결과 붙여넣기]

출력 형식:
# 최종 코드 리뷰 리포트

## 1. 코드 구조 요약 (Sub A 기반)
## 2. 보안 이슈 (Sub B 기반, 심각도 순 정렬)
## 3. 스타일 이슈 (Sub C 기반)
## 4. 수정 우선순위 (HIGH → MEDIUM → LOW)
## 5. 종합 평가 (/10)
```

## 병렬 실행 확인 포인트

### 1. 역할 격리 확인
각 에이전트가 자신의 역할 외 내용을 출력하지 않는지 확인한다.

| 에이전트 | 해야 하는 것 | 하지 말아야 할 것 |
|---------|------------|-----------------|
| Sub A | 구조 설명 | 버그 언급, 스타일 평가 |
| Sub B | 취약점 나열 | 구조 설명, 수정 코드 제시 |
| Sub C | 스타일 위반 | 버그 분석, 구조 설명 |

### 2. 독립성 확인
- Sub B가 Sub A의 결과 없이도 완전한 분석을 했는가?
- Sub C가 Sub B의 결과 없이도 완전한 분석을 했는가?
- 세 에이전트의 결과 순서가 달라도 최종 리포트에 영향이 없는가?

### 3. 오케스트레이터 역할 확인
- 새로운 버그를 발견하지 않고 통합만 했는가?
- 3개 에이전트의 내용이 모두 반영되었는가?

## 핵심 학습 포인트

```
서브에이전트 병렬 패턴의 필수 조건:

1. 공통 입력   — 3개 에이전트 모두 원본 코드를 직접 입력받음
2. 무의존성    — 에이전트 간 결과를 주고받지 않음
3. 역할 분리   — 각자 다른 관점에서 분석
4. 후처리 통합 — 오케스트레이터가 마지막에 조합
```

이 패턴을 실무에 적용하면:
- 대용량 코드베이스를 모듈별로 나눠 병렬 분석
- 같은 문서를 다양한 관점(보안/성능/가독성)으로 동시 검토
- 여러 파일을 동시에 처리해서 전체 시간 단축
