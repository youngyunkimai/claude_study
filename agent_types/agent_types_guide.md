# 서브에이전트 vs 팀에이전트 완전 정리

## 1. 개념 정의

### 서브에이전트 (Sub-agent)

메인 에이전트(오케스트레이터)가 특정 작업을 위해 **별도로 생성하는 독립 에이전트**.

- 독립적인 컨텍스트 창을 가짐 → 메인 컨텍스트 절약
- 작업 완료 후 결과만 메인에 반환
- 병렬 실행 가능
- 특화된 도구 세트만 부여 가능
- 단방향 통신 (오케스트레이터 → 서브에이전트 → 결과 반환)

```
메인 에이전트 (오케스트레이터)
  ├── Sub A: 결과 반환
  ├── Sub B: 결과 반환
  └── Sub C: 결과 반환
```

### 팀에이전트 (Team agent / Multi-agent)

여러 에이전트가 **협력하여 하나의 큰 작업을 분담**하는 구조. 에이전트 간 양방향 통신 가능.

```
오케스트레이터
  ├── Worker A ←──→ Worker B
  └── Worker C ←──→ Worker A
```

---

## 2. 장단점 비교

| 항목 | 서브에이전트 | 팀에이전트 |
|------|------------|-----------|
| **컨텍스트 비용** | 낮음 (격리) | 높음 (공유/누적) |
| **속도** | 병렬 실행 가능, 빠름 | 통신 오버헤드 있음 |
| **구현 복잡도** | 낮음 | 높음 |
| **에러 전파** | 격리됨 (독립 실패) | 전파 가능 |
| **상호 협력** | 불가 (단방향) | 가능 (양방향) |
| **작업 수정** | 오케스트레이터만 가능 | 에이전트끼리 피드백 |
| **일관성 유지** | 오케스트레이터가 보장 | 합의 필요, 복잡 |
| **디버깅** | 쉬움 | 어려움 |

### 컨텍스트 비용 차이 예시

```
팀에이전트 (공유 컨텍스트):
  Agent A → 10,000 토큰 결과
  Agent B → 10,000 토큰 결과
  Agent C → 10,000 토큰 결과
  ──────────────────────────
  오케스트레이터 컨텍스트: 30,000 토큰 누적 → 비용 폭발, 느려짐

서브에이전트 (격리된 컨텍스트):
  Agent A → 요약 500 토큰만 반환
  Agent B → 요약 500 토큰만 반환
  Agent C → 요약 500 토큰만 반환
  ──────────────────────────
  오케스트레이터 컨텍스트: 1,500 토큰만 누적
```

## 3. 선택 기준

```
에이전트 간 대화가 필요한가?
  YES → 팀에이전트
  NO  → 서브에이전트로 충분

작업이 독립적인가?
  YES → 서브에이전트 (병렬 실행)
  NO  → 팀에이전트 or 순차 서브에이전트

컨텍스트 비용이 중요한가?
  YES → 서브에이전트
  NO  → 팀에이전트 가능
```

## 4. 서브에이전트 사용 예시 (충분한 경우)

### 코드베이스 병렬 분석

```
오케스트레이터: "각자 맡은 모듈 분석해와"
  ├── Sub A: auth 모듈 분석    → 요약 반환
  ├── Sub B: network 모듈 분석 → 요약 반환
  └── Sub C: storage 모듈 분석 → 요약 반환

→ 병렬 실행, 빠름, 저렴, 단순
→ 에이전트끼리 소통할 필요 없음
```

## 5. 팀에이전트가 반드시 필요한 예시

핵심 조건: **에이전트 A의 출력이 에이전트 B의 입력에 영향을 주고, B의 결과가 다시 A를 바꿔야 할 때**

### 예시 1: 자동 코드 리뷰 + 수정 루프

```
Coder Agent          Reviewer Agent
    │                     │
    │── 코드 작성 ────────>│
    │                     │── "null 체크 없음, 버그"
    │<── 수정 요청 ────────│
    │── 수정된 코드 ──────>│
    │                     │── "OK, LGTM"
    │<── 승인 ─────────────│
```

Reviewer의 피드백을 받아서 Coder가 재작업해야 하므로, 왕복 통신이 N회 반복. 서브에이전트는 단방향이라 이 루프를 구현할 수 없음.

### 예시 2: 보안 Red Team vs Blue Team 시뮬레이션

```
Red Team Agent           Blue Team Agent
    │                         │
    │── SQL Injection 시도 ───>│
    │                         │── 패치 적용
    │<── 패치된 코드 ───────────│
    │── 우회 공격 재시도 ──────>│
    │                         │── 추가 방어
    │<── ...반복... ────────────│
```

Red의 공격 결과를 보고 Blue가 방어를 수정하고, 그 방어를 보고 Red가 전략을 바꾸는 **상호 적응(adaptive)** 구조.

### 예시 3: LLM 기반 토론/합의 시스템

```
Agent A (낙관론자)    Agent B (비관론자)    Judge Agent
      │                    │                   │
      │── "이 설계 좋아" ───>│                   │
      │                    │── "성능 문제 있어" ─>│
      │<── B 주장 전달 ──────────────────────────│
      │── "반론: 캐싱으로 해결" ─────────────────>│
      │                    │── "그럼 메모리 문제" >│
      │                    │                   │── 최종 판단
```

여러 에이전트가 서로의 주장을 보고 입장을 동적으로 수정하며 합의를 도출.

### 예시 4: 실시간 Pair Programming

```
Driver Agent              Navigator Agent
    │                          │
    │── "auth 함수 작성중" ─────>│
    │                          │── "잠깐, JWT 방식 바꿔야 해"
    │<── 설계 변경 지시 ─────────│
    │── "변경했어, 계속할게" ────>│
    │                          │── "다음은 refresh token 구현해"
```

Navigator의 지시 내용이 Driver의 현재 상태에 따라 달라지기 때문에 서브에이전트로는 표현 불가.

---

## 6. 팀에이전트가 반드시 필요한 패턴 정리

| 패턴 | 설명 |
|------|------|
| **피드백 루프** | A 결과 → B 판단 → A 재작업 (N회 반복) |
| **상호 적응** | 상대방 행동을 보고 전략 변경 |
| **합의 도출** | 여러 에이전트가 토론하며 결론 수렴 |
| **실시간 조율** | 한 에이전트의 현재 상태가 다른 에이전트의 다음 지시를 결정 |

> **한 문장 요약:** "이전 결과를 보고 나서야 다음 지시를 결정할 수 있는 구조"라면 팀에이전트가 필요하다.

---

## 7. 커널 개발자를 위한 비유

| | 서브에이전트 | 팀에이전트 |
|--|------------|-----------|
| **프로세스 비유** | `fork()` + `wait()` | IPC(pipe, socket) 사용하는 프로세스 협력 |
| **병렬 처리 비유** | kthread (부모가 생성, 결과 회수) | SMP에서 각 CPU 코어가 독립적으로 협력 처리 |
| **특징** | 단순하고 격리됨 | 강력하지만 복잡도와 오버헤드 증가 |

---

## 결론

팀에이전트는 에이전트 간 상호작용이 **반드시 필요한 경우에만** 사용하는 것이 원칙.
나머지 대부분의 경우엔 서브에이전트가 더 빠르고, 저렴하고, 디버깅도 쉽다.
